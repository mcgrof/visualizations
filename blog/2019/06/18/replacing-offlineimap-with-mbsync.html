Replacing offlineimap with mbsync

original source ↗

The Problem with offlineimap

The author had been using offlineimap for email synchronization but encountered persistent memory issues. Upon reviewing system logs and utilization graphs, offlineimap was consistently running out of memory (OOM) during every sync cycle. This posed risks including potential filesystem corruption and system instability.

Why mbsync?

The switch was made to mbsync, described as "written in C, actively maintained and has mutt code pedigree." The tool uses a channel-based architecture where each channel has a master (remote) and local store for bidirectional synchronization.

Configuration for Gmail Nested Labels

A practical configuration example demonstrates how to flatten Gmail's nested label structure (like "linux-kernel/linux-xfs") into local maildir folders. Key settings include:

  • Application-specific passwords required for Gmail access
  • Throttling via PipelineDepth to avoid exceeding Gmail's IMAP quotas
  • Channel grouping for organized syncing

Systemd Integration

Rather than using cron, the implementation uses per-channel systemd timer and service units, enabling:

  • Flexible scheduling (e.g., 1-minute intervals for urgent mail, 30-minute intervals for mailing lists)
  • Centralized logging via journalctl
  • Sequential execution preventing resource conflicts

Results

mbsync demonstrated dramatically better memory efficiency. Running five concurrent channels produced stable, predictable resource usage without OOM incidents -- a stark improvement over offlineimap's behavior.
